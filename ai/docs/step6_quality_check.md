# Step 6: 품질 검증 강화 (Quality Check)

## 목표
Step 4에서 기본 구현한 `quality_check` 노드를 강화한다.
검증 규칙을 확정하고, 재시도 시 실패 이유가 프롬프트에 반영되도록 `llm_report.py`를 업데이트한다.

---

## 수정 파일

```
ai/report/nodes/
├── quality_check.py    ← 규칙 강화
└── llm_report.py       ← 재시도 프롬프트 개선
```

---

## `quality_check.py` 강화 규칙

Step 4에서 정의한 3가지 규칙을 확정하고 엣지 케이스를 처리한다.

| 규칙 | 판단 기준 | 실패 시 `quality_issues` 메시지 |
|------|-----------|-------------------------------|
| 최소 길이 | `len(report) >= 100` | "리포트가 너무 짧습니다 (N자)" |
| bullet 항목 | `-, •, *, 번호목록` 패턴 존재 | "개선 방향 bullet 항목이 없습니다" |
| 숫자 인용 | `\d+` 패턴 존재 | "통계 숫자가 인용되지 않았습니다" |

**선택적 규칙 (구현 복잡도와 상의하여 결정)**:
- 알려진 `category_stats` 카테고리가 리포트에 전혀 언급되지 않으면 환각 의심 경고

---

## `llm_report.py` 업데이트

재시도 시(`retry_count > 0`) 프롬프트 끝에 이전 실패 이유를 추가한다.

```
...기존 프롬프트...

이전 시도에서 다음 문제가 있었습니다:
- {quality_issues[0]}
- {quality_issues[1]}
위 문제를 반드시 수정하여 다시 작성하세요.
```

이 방식으로 모델이 같은 실수를 반복하는 것을 억제한다.

---

## 주의사항

- `quality_check`는 동기 함수 (LLM 호출 없음)
- 규칙이 너무 엄격하면 최대 재시도(2회)를 모두 소진하고 미달 리포트를 반환할 수 있음
  → 100자 기준은 Qwen2.5 3B의 실제 응답 길이를 확인한 후 조정 필요
- `quality_issues`가 비어있으면 `quality_passed=True`
